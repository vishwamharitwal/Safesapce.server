# WebRTC Deep Fix Plan

## Summary
30 issues found: 6 Critical, 11 High, 10 Medium, 3 Low
Fixing all Critical + High issues. Medium/Low where feasible without UI changes.

---

## Fix 1: `_closeWebRTC()` doesn't await async disposal (A1)
**File**: `signaling_service.dart:732-766`

Problem: `peerConnection.close()`, `peerConnection.dispose()`, `localStream.dispose()`, `remoteStream.dispose()` all return Futures but are NOT awaited. Native resources leak.

Fix: Make `_closeWebRTC()` async, await all disposal calls, add guard against double-call.

---

## Fix 2: WebRTC init error silently swallowed (A2)
**File**: `signaling_service.dart:685-690`

Problem: `completer.complete()` always succeeds even when init fails. Callers think init worked but `peerConnection` is null.

Fix: Use `completer.completeError(e)` in catch block. Verify `peerConnection != null` after awaiting.

---

## Fix 3: Remove 500ms delay causing ICE race (A3)
**File**: `signaling_service.dart:242-245`

Problem: 500ms sleep after `_initWebRTC()` allows ICE candidates to queue up and potentially overflow.

Fix: Remove the delay. Send offer immediately after init completes.

---

## Fix 4: `_isInit` flag prevents socket re-creation (B1)
**File**: `signaling_service.dart:143-146, 723-730`

Problem: After `disconnect()`, `_isInit = false` but old socket listeners remain. New `connect()` creates new socket but old one leaks.

Fix: Call `socket.clearListeners()` and `socket.destroy()` in `disconnect()` before creating new socket.

---

## Fix 5: Singleton state leak between sessions (B2)
**File**: `signaling_service.dart:64-72, 732-766`

Problem: `partnerId`, `partnerName`, `partnerAvatar`, `partnerRating` not reset in `_closeWebRTC()`.

Fix: Reset ALL session-scoped fields in `_closeWebRTC()`.

---

## Fix 6: Triple navigation race in matchmaking (C5)
**File**: `matchmaking_screen.dart:46-179`

Problem: 3 independent paths navigate to ActiveSessionScreen: `onMatchFound` (500ms delay), `onPartnerConnected` (800ms delay), polling timer (500ms interval).

Fix: Set `_hasMatched = true` SYNCHRONOUSLY when any trigger fires, BEFORE any async operation. Remove 500ms fallback in `onMatchFound`.

---

## Fix 7: Triple navigation in partner_preview (C11)
**File**: `partner_preview_screen.dart:122-175`

Problem: Same issue - 3 paths navigate.

Fix: Set `_hasNavigated` flag synchronously in all paths.

---

## Fix 8: ICE candidates added without await (A4)
**File**: `signaling_service.dart:437-448`

Problem: `addCandidate` called in loop without await. Candidates may be processed out of order.

Fix: Use `await` in the loop.

---

## Fix 9: Callbacks not cleared across screens (B3, B7, B9, B12)
**Files**: `active_session_screen.dart`, `incoming_call_screen.dart`, `chat_room_screen.dart`

Problem: Each screen sets callbacks but doesn't clear all of them on dispose.

Fix: Each screen's `dispose()` calls `signalingService.clearCallbacks()` or clears ALL callbacks it set.

---

## Fix 10: Double `leaveSession` in active_session (C3)
**File**: `active_session_screen.dart:167-201`

Problem: `_endSession()` calls `leaveSession()`, then `dispose()` also calls `leaveSession()`.

Fix: Add `_sessionEnded` flag. Skip `leaveSession()` in dispose if already ended.

---

## Fix 11: Singleton state leak in active_session_screen (C3)
**File**: `active_session_screen.dart:192-201`

Problem: `dispose()` calls `leaveSession()` which calls `_closeWebRTC()` which resets ALL signaling state. But `PostSessionScreen` needs `partnerId` from signaling.

Fix: Capture `partnerId` BEFORE calling `leaveSession()`.

---

## Fix 12: WebRTC state not reset on skip-match (C7)
**File**: `matchmaking_screen.dart:181-198`

Problem: `onMatchSkipped` calls `findMatch` again but doesn't clean up previous WebRTC state.

Fix: Call `_closeWebRTC()` before re-finding match.

---

## Fix 13: `onMatchFound` in chat_room causes unexpected navigation (B11)
**File**: `chat_room_screen.dart:129-146`

Problem: `onMatchFound` registered in `initState` can fire unexpectedly from other screens.

Fix: Only register callback when call feature is actually used (not in initState).
